# Command-line syntax in Katalon Runtime Engine

This guide lists out the supported command-line syntax and arguments of Katalon Runtime Engine.


:::note

-   From version 7.9.0 onwards, you can upgrade the default JRE 8 to higher versions. You can refer to this document for further details: [Run tests with another JRE in the command line](/get-started/set-up-your-workspace/katalon-studio-preferences/set-a-new-default-jre-for-test-projects-in-katalon-studio#).

:::

## Requirements

-   Katalon Runtime Engine is installed. See: [Get started with Katalon Runtime Engine](/execute/katalon-runtime-engine/get-started-with-katalon-runtime-engine).
-   A Katalon Runtime Engine License. To learn more about activating your license, you can refer to this document: [Katalon license](/administer/katalon-studio-enterprise-and-katalon-runtime-engine-license/activate-a-katalon-license).

## Use plugins in console mode

You can also continue using your plugins of choice with console commands. See this guide: [Use Plugins in Console Mode](/plugins-and-add-ons/katalon-store/katalon-studio-plugins/using-katalon-store-plugins#).

## General arguments

Here's the list of options supported for the <code>katalonc</code> commands for Katalon Studio version 7.0.0 onwards.

### Required arguments

-   **<code><strong>-runMode</strong>=console</code>**

    Enable console mode.

-   **<code><strong>-projectPath</strong>=&lt;path&gt;</code>**

    Specify the project location \(include .prj file\). The absolute path must be used in this case.

-   **<code><strong>-testSuitePath</strong>=&lt;path&gt;</code>**

    Specify the test suite file \(without extension .ts\). The relative path \(root being project folder\) must be used in this case.

-   **<code><strong>-testSuiteCollectionPath</strong>=&lt;path&gt;</code>**

    Specify the test suite file \(without extension .tsc\). The relative path \(root being project folder\) must be used in this case.

    
    :::note

    -   If <code>-testSuitePath</code> is specified, <code>-testSuiteCollectionPath</code> is not required.

    :::

-   **<code><strong>-browserType</strong>=&lt;browser&gt;</code>**

    Specify the browser type used for test suite execution.

    From version 7.6+, you can use this option in Test Suite Collection execution. The specified browser is used for all test suites in that collection.

    The following browsers are supported in KRE:

    -   Firefox
    -   Chrome
    -   IE
    -   Edge
    -   Edge \(Chromium\)
    -   Safari
    -   Remote
    -   Android
    -   iOS
    -   Web Service
    
    :::note

    -   Only Chrome, Firefox, and Remote are available for use in the Linux version.

        **<code>Web Service</code> is used for Web Service test execution.**

    :::

-   **<code><strong>-apiKey</strong>=&lt;Your&#95;API&#95;Key&gt;</code> or <code><strong>-apikey</strong>=&lt;Your&#95;API&#95;Key&gt;</code>**

    API Keys are used to represent a user's credentials. See: [API Keys](/administer/settings/katalon-api-key-in-katalon-testops).


### Optional arguments

-   **<code><strong>-statusDelay</strong>=&lt;seconds&gt;</code>**

    System updates execution status of the test suite after the delay period \(in seconds\) specified.

-   **<code><strong>-retry</strong>=&lt;number of retry times&gt;</code>**

    Number of times running test cases in the test suite.

-   **<code><strong>-retryStrategy</strong>=&lt;allExecutions | failedExecutions | immediately&gt;</code>**

    This option is supported in version 7.6 onwards. Specify which execution to be retried \(this parameter overrides setting in test suite file\):

    -   **allExecutions**: Retry all executions when the Test Suite fails
    -   **failedExecutions**: Retry only failed executions when the Test Suite fails
    -   **immediately**: Retry a failed execution of a test case or test data immediately. \(Only for Katalon Studio Enterprise users\)
-   **<code><strong>-maxFailedTest</strong>s=&lt;T&gt;</code>**

    -   From version 8.1.0, you can terminate a test suite/ test suite collection execution based on the number of test failures.
    -   Set &lt;T&gt; as the maximum number of total test failures allowed in the execution. Reaching &lt;T&gt; terminates the test execution.
    -   A test failure is counted when any of these type of tests fails: test case, retried test case, test iteration, or retried test iteration.
-   **<code><strong>-reportFolde</strong>r=&lt;path&gt;</code>**

    Specify the destination folder for saving report files. You can use an absolute path or relative path \(root being project folder\).

-   **<code><strong>-reportFileName</strong>=&lt;name&gt;</code>**

    Specify the name for report files \(.html, .csv, .log\). If not provided, the system uses the name "report" \(report.html, report.csv, report.log\). .

    
    :::note

    -   <code>-reportFileName</code> is only interpreted if <code>-reportFolder</code> is specified.

    :::

-   **<code><strong>-sendMail</strong>=&lt;e-mail address&gt;</code>**

    Specify the e-mail address for receiving report files. If the e-mail address was not specified, the report files are not to be sent.

-   **<code><strong>-remoteWebDriverUrl</strong>=&lt;remote web server url&gt;</code>**

    Specify the remote web driver URL.

-   **<code><strong>-serverUrl</strong>=&lt;server url&gt;</code>**

    Specify the server URL that you use to activate a KRE license. From Katalon Studio version 8.3.5 onwards, you can use this parameter to activate a KRE license with Private Instance while executing in console mode. For example: <code>-serverUrl="https://admin-tenant1.katalon-cloudops.com"</code>.

-   **<code><strong>-remoteWebDriverType</strong>=&lt;Selenium | Appium&gt;</code>**

    Remote web's driver type.

    
    :::note

    -   <code>remoteWebDriverType</code> is only interpreted if <code>remoteWebDriverUrl</code> is specified.

    :::

-   **<code><strong>-deviceId</strong>=&lt;device Id for Android/device UUID for ios&gt;</code>**

    Specify the device's ID to execute test scripts using this device.

    
    :::note

    -   <code>-deviceId</code> is only interpreted if <code>-browserType</code> is specified.
    :::

-   **<code><strong>-executionProfile</strong></code>**

    Specify the execution profile that a test suite executes with

    From version 7.6.0 onwards, you can use this option in Test Suite Collection execution. The specified execution profile is applied to all test suites in that collection.

-   **<code><strong>-delayBetweenInstances</strong>=&lt;value&gt;</code>**

    -   From Katalon version 8.2.0 onwards, you can set a delay between each test suite execution in a Test Suite Collection.
    -   <code>Value</code> is a positive integer from 0-999 seconds.
    -   When a test suite is ready to start, KRE prints a message in Console Log: <code>Test suite $&#10100;testSuiteID&#10101; is ready to start at $&#10100;currentTimeStamp&#10101;</code>
-   **<code><strong>-g&#95;&lt;variable&#95;name&gt;</strong></code>**

    Override Execution Profile variables.

    Example:<code>-g&#95;userName="admin"</code>

-   **<code><strong>-testSuiteQuery</strong>=&lt;search-query&gt;</code>**

    From version 7.8.0, you can override the search query of the dynamic test suite from the CLI.

    <code>&lt;search-query&gt;</code> is the search condition that follows the query syntax of the installed query provider plugin. For detailed instruction, follow this guide: [Pass the search query to the CLI for dynamic test suite execution](/execute/katalon-runtime-engine/create-dynamic-test-suite-at-runtime#).

    Example:<code>-testSuiteQuery="ids=(Test Cases/TC1&#95;Verify Successful Login,Test Cases/TC2&#95;Verify Successful Appointment)"</code>

-   **<code><strong>-testSuiteCollectionQuery</strong></code>**

    From version 8.0.0, you can enable or disable Test Suite\(s\) in Test Suite Collection.

    Example:<code>-testSuiteCollectionQuery="indexes=(1,3)"</code>

-   **<code><strong>-maxResponseSize</strong></code>**

    Override the maximum response size in project setting \(available from version 7.6\). Learn more about [Web Service Settings](/create-tests/manage-projects/project-settings/set-desired-execution-behaviors-of-katalon-studio#).

    Example:<code>-maxResponseSize=400</code>

-   **<code><strong>-licenseRelease</strong> <strong>-orgID</strong>=&lt;organization's id&gt;</code>**

    From version 8.0.0, you can release the previous execution session before checking the license.

    Example:

    <code>-licenseRelease=true</code>

    <code>-orgID=89151</code>

-   **<code><strong>--config</strong> <strong>-webui.autoUpdateDrivers</strong>=true</code>**

    Allow WebDriver binaries to be updated automatically in console mode.


### Windows-specific arguments

-   **<code><strong>-consoleLog</strong></code>**

    Display log in the console. Only use this option when running Katalon Studio in Windows Command Prompt. Do not use this option in other OSes or CI tools, for example Jenkins.

-   **<code><strong>-noExit</strong></code>**

    Keep the console open after the execution is completed. Only use this option when running Katalon Studio in Windows Command Prompt. Do not use this option in other OSes or CI tools, for example Jenkins.


## Proxy arguments


:::note

-   From version 7.5.0 onwards, there are two types of proxy configurations: authentication and system proxies. To learn more about configuring different proxy preferences, you can refer to this document: [Proxy Preferences](/get-started/set-up-your-workspace/katalon-studio-preferences/configure-proxy-preferences-in-katalon-studio) for further details.

-   From version 7.2.0 onwards, you can exclude a list of hosts from proxy in the **Manual proxy configuration**. To learn more about manually excluding domains from proxy, you can refer to this document: [Proxy settings](/get-started/set-up-your-workspace/katalon-studio-preferences/preferences-in-katalon-studio).

-   From version 7.0.0 onwards, you can pass proxy details via a request object in Web Service testing. To learn more about passing proxy details in script mode, you can refer to this document: [Override proxy details in the test script](/get-started/set-up-your-workspace/katalon-studio-preferences/preferences-in-katalon-studio).


:::

These proxy options must be used with <code>--config</code> parameter, for example: <code>--config -proxy.auth.option=MANUAL&#95;CONFIG</code>

```
katalonc -noSplash -runMode=console -projectPath="C:\Users\Katalon Studio\Project\YourProject.prj" -retry=0 -testSuitePath="Test Suites/download" -executionProfile="default" -browserType="Chrome" --config -proxy.auth.option=MANUAL_CONFIG -proxy.auth.server.type=HTTP -proxy.auth.server.address=192.168.1.16 -proxy.auth.server.port=16000 -proxy.system.option=MANUAL_CONFIG -proxy.system.server.type=HTTP -proxy.system.server.address=127.0.0.1 -proxy.system.server.port=12701 -proxy.system.username=katalon -proxy.system.password=T3stP@zZW0rol -proxy.system.applyToDesiredCapabilities=true

```

### Authentication proxy arguments

-   **<code><strong>-proxy.auth.option</strong></code>**

    The available values for this option are: <code>NO&#95;PROXY</code>, <code>USE&#95;SYSTEM</code>, or <code>MANUAL&#95;CONFIG</code>.

-   **<code><strong>-proxy.auth.server.type</strong></code>**

    The available server types are: <code>HTTP</code>, <code>HTTPS</code>, or <code>SOCKS</code>.

-   **<code><strong>-proxy.auth.server.address</strong></code>**

    Example: locahost, katalon.com.

-   **<code><strong>-proxy.auth.server.port</strong></code>**

    Example: 80, 8080, 9999.

-   **<code><strong>-proxy.auth.excludes</strong></code>**

    Exclude the specified server.

    For example: 127.0.0.1, localhost, myserver.com.

-   **<code><strong>-proxy.auth.username</strong></code>**

    Example: MyProxyUsername. This option is required if the specified proxy server requires authentication.

-   **<code><strong>-proxy.auth.password</strong></code>**

    Example: MyProxyPassword. This option is required if the specified proxy server requires authentication.


### System proxy arguments

**For versions 7.5.0 onwards**

-   **<code><strong>-proxy.system.option</strong></code>**

    The available values for this options are: NO\_PROXY, USE\_SYSTEM, or MANUAL\_CONFIG.

-   **<code><strong>-proxy.system.server.type</strong></code>**

    HTTP, HTTPS, or SOCKS

-   **<code><strong>-proxy.system.server.address</strong></code>**

    Example: locahost, katalon.com.

-   **<code><strong>-proxy.system.server.port</strong></code>**

    Example: 80, 8080, 9999.

-   **<code><strong>-proxy.system.excludes</strong></code>**

    Example: 127.0.0.1, localhost, myserver.com.

-   **<code><strong>-proxy.system.username</strong></code>**

    Example: MyProxyUsername. This option must be specified if your proxy requires authentication.

-   **<code><strong>-proxy.system.password</strong></code>**

    Example: MyProxyPassword. This option must be specified if you proxy requires authentication.

-   **<code><strong>-proxy.system.applyToDesiredCapabilities</strong></code>**

    Example: true or false.


**For versions below 7.5.0**

-   **<code><strong>-proxy.option</strong></code>**

    NO\_PROXY, USE\_SYSTEM, or MANUAL\_CONFIG.

-   **<code><strong>-proxy.server.type</strong></code>**

    HTTP, HTTPS, or SOCKS.

-   **<code><strong>-proxy.server.address</strong></code>**

    Example: locahost, katalon.com.

-   **<code><strong>-proxy.server.port</strong></code>**

    Example: 80, 8080, 9999.

-   **<code><strong>-proxy.excludes</strong></code>**

    Example: 127.0.0.1, localhost, myserver.com.

-   **<code><strong>-proxy.username</strong></code>**

    Example: MyProxyUsername. This option must be specified if your proxy requires authentication.

-   **<code><strong>-proxy.password</strong></code>**

    Example: MyProxyPassword. This option must be specified if your proxy requires authentication.


## Integration arguments

### TestOps integration arguments

-   **<code><strong>--info</strong> <strong>-buildLabel</strong>=&lt;text&gt; <strong>-buildURL</strong>=&lt;text&gt;</code>**

    Pass the build's label and URL, which are displayed in Katalon TestOps.

    Example:

    ```
    --info -buildLabel="Build 1" -buildURL="http://192.168.35.52:8080/job/katalon-demo/job/master/179/"
    ```

-   **<code><strong>-testOpsBuildId</strong></code>**

    From version 8.0.0, you can specify the build ID to update the Test Suite/Test Suite Collection report.

    Example:<code>-testOpsBuildId=24</code>


-   **<code><strong>-testOps.baselineCollectionId</strong>=&lt;CollectionId&gt;</code>**

    From version 8.4.0 onwards, you can specify a baseline collection ID to which Katalon visual testing functionality compares newly captured images in your test runs. See: [Use TestOps Visual Testing](/analyze/analytics/visual-testing/use-testops-visual-testing).

    For example:

    ```
    -testOps.baselineCollectionId=1234
    ```

-   **<code><strong>-testOps.serverUrl</strong>=&lt;URL&gt;</code>**

    From version 8.4.0 onwards, you can specify the URL for TestOps server for authentication and integration. You can use this argument to work with different offerings of TestOps, including:

    -   Public Cloud and On-premise.

    -   Private Instance.

    -   Staging and Production.

    
    :::note

    -   If <code>-testOps.serverUrl</code> is not specified, the default URL is <code>https://testops.katalon.io</code>.

    :::


-   **<code>-testOps.customField=&lt;key1:value1, key2:value2&gt;</code>**

    Assigns pre-existent custom fields to the test run. Multiple custom fields can be assigned, separated by a comma. The command also creates a custom field and assign it to the test run if the custom field was never created before. To understand more about custom field, see [Custom Fields and Tags](/organize/custom-field-and-tags/custom-fields-and-tags).

-   **-testOps.tag=&lt;tag1, tag2&gt;**

    Assign pre-existent tags to the test run. Multiple tags can be assigned, separated by a comma. The command also creates a tag and assign it to the test run if the tag was never created before. To understand more about tag, see [Custom Fields and Tags](/organize/custom-field-and-tags/custom-fields-and-tags).


### TestCloud integration arguments

-   **<code><strong>-testcloudEnvironmentId</strong>=&lt;ID&gt;</code>**

    The ID of the environment which corresponds to a combination of OS, browser type and browser version to execute \(Available from 8.2.5 onwards\).

    
    :::note

    -   To get the TestCloud environment ID, open Katalon Studio and use the command builder. For detailed information, follow this guide: Generate Command with Command Builder.

    -   This parameter already contains the information about browser type. Therefore, the <code>-browserType</code> parameter is not generated in this integration.

    :::

-   **<code><strong>-testcloudTunnel</strong></code>**

    Allow the execution to be performed via a tunnel \(Available from 8.2.5 onwards\).


### Kobiton integration arguments

-   **<code><strong>--config</strong> <strong>-kobiton.authentication.username</strong>=&lt;yourKobitonUsername&gt; <strong>-kobiton.authentication.password</strong>=&lt;yourPassword&gt;</code>**

    Passing Kobiton username and password.

-   **<code><strong>--config</strong> <strong>-kobiton.authentication.serverUrl</strong>=&lt;defaultKobitonServer&gt; <strong>-kobiton.authentication.username</strong>=&lt;yourKobitonUsername&gt; <strong>-kobiton.authentication.apiKey</strong>=&lt;yourKobitonAPIKey&gt;</code>**

    Passing Kobiton Server URL, username, and APIKey \(Available in 7.8 and later\).

-   **<code><strong>-kobitonDeviceId</strong>=&lt;yourKobitonDeviceId&gt;</code>**

    ID of the destination where the result is uploaded on it.


### Azure DevOps integration arguments

-   **<code><strong>-adoPlanId</strong>=&lt;testplanID&gt;</code>**

    ID of the test plan used for submitting test run\(s\)\(available from version 8.0.0\).

-   **<code><strong>-adoTestRunName</strong>=&lt;testRunName&gt;</code>**

    From version 8.0.0, you can create test run\(s\) on ADO with the specified name.

-   **<code><strong>--info</strong> <strong>-adoDefinitionID</strong>=&lt;definitionID&gt;</code>**

    From version 8.0.0, you can get the latest completed Build ID of the specified Definition ID and pass it to Test Run properties on ADO.
